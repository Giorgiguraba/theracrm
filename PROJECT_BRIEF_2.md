# PROJECT BRIEF — TheraCRM (Therapy Center CRM, SaaS model)

> Handoff document. Read this fully before writing any code. The goal is a lean,
> modern CRM for a small psychology/therapy center in Georgia (country), built and
> operated as a SaaS by the vendor (Gio). Vendor keeps full platform control:
> support access, subscription management, and the ability to suspend tenant
> access on payment delay.

---

## 1. Client profile (from discovery call, June 2026)

- **Business**: Psychology center — individual therapy, art therapy, internship
  program for psychology students, multiple therapy types / courses.
- **Customers**: B2C individuals (mostly psychology students and people
  interested in the field). B2B/corporate possible later — design for it, don't
  build it yet.
- **Team**: 2 users now (owner + one operator who calls leads). Must be easy to
  add a 3rd+ user later.
- **Lead volume**: ~40–50 leads/month, generated by $200–300/month Facebook +
  Instagram ad spend.
- **Current process**: Leads come from FB/Insta → moved to chat after
  registration → written down in a paper notebook. No digital system at all.
- **Main pain**: Leads say "I'm thinking about enrolling" and then everyone
  forgets about them — both the lead and the operator. Follow-up is the #1
  feature, not a nice-to-have.
- **Pipeline**: Very simple. "Interested → enrolled". A lead may message, think
  for a few days, then decide. No multi-stage B2B sales process.
- **Email automation**: Wanted ("you are registered" confirmations, reminders).
- **Reporting**: Helpful but not critical for v1. Owner wants to see whether the
  ad budget converts.
- **Maturity level**: LOW. Coming from a paper notebook. UX must be dead simple.
  One 30-minute onboarding session planned. If the UI is complicated, the
  product dies — this killed CRMs for them before it even started.

## 2. Product decisions

- **Language of UI**: Georgian (ka) primary, English (en) secondary. Use i18n
  from day one (next-intl), all strings in locale files.
- **Mobile-friendly is mandatory** — operator will often work from a phone.
- **Simplicity over features.** Every screen must pass the test: "could a person
  who used a paper notebook yesterday use this without training?"

## 3. Architecture — two applications, one platform

### App A — Tenant CRM (what the therapy center uses)
Modules (MVP order):
1. **Leads inbox** — auto-captured from Meta Lead Ads webhook + manual add
   (for people who write in chat/DM). Fields: name, phone, email, source
   (fb_ads / ig_ads / manual / referral), program of interest, notes.
2. **Kanban pipeline** — 4 columns: New → Contacted → Thinking → Enrolled /
   Lost. Drag & drop. Realtime sync between users (Supabase Realtime).
3. **Follow-up engine** — if a lead sits in "Thinking" > 48h (configurable):
   (a) in-app + email reminder to the operator, (b) optional automated email to
   the lead from a template. Snooze / done actions.
4. **Email automation** — templates: enrollment confirmation, course start
   reminder, follow-up nudge. Provider: Brevo (free tier = 300 emails/day,
   far above needs). Template variables: {name}, {program}, {start_date}.
5. **Programs/services catalog** — therapy types, courses, internship cohorts.
   Each lead links to a program. Program has: name, price, start date, capacity.
6. **Dashboard** — leads this month, conversion rate by stage, leads by source,
   simple ad ROI view (ad spend input vs enrolled × program price).
7. **Activity log per lead** — calls, notes, emails sent, stage changes.
   Timestamped, per-user.
8. **Users & roles (tenant-level)** — Admin (owner) / Operator. Admin manages
   tenant users, templates, programs. Operator works leads.

### App B — Platform Super Admin (vendor's control panel)
Separate route group `/platform`, separate role, invisible to tenants.
1. **Tenant management** — list of client organizations, plan, status
   (active / past_due / suspended / cancelled), created date, user count.
2. **Subscription & billing control** — manual for v1: vendor marks invoices
   paid/unpaid. `paid_until` date per tenant. Automatic status flow:
   - `paid_until` passed → status `past_due`, banner shown in tenant UI:
     "Payment due — service will be limited on <date>".
   - +7 days grace (configurable) → status `suspended`: middleware blocks all
     tenant routes, shows a payment-required screen with vendor contact.
     **Data is never deleted** — suspension is a read-block, reactivation is
     instant on payment.
3. **Kill-switch / reactivate** — one click suspend/restore per tenant.
4. **Support impersonation** — "Login as tenant admin" (audited!) to reproduce
   and fix issues without asking for screenshots.
5. **Audit log** — all platform-admin actions logged (who, what, when).
6. **Feature flags per tenant** — enable/disable modules (e.g. corporate/B2B
   module later as a paid upsell).
7. **Health overview** — webhook delivery failures, email send failures, error
   counts per tenant.

### Multi-tenancy strategy
- Single Postgres database, `tenant_id` column on every tenant table.
- **Supabase RLS (Row Level Security) enforces isolation** — every policy
  checks `tenant_id = auth.jwt() -> app_metadata -> tenant_id`. This is the
  core security mechanism; never query tenant tables without RLS on.
- Platform admin uses service-role access through server-side code only.
- Even though there is 1 client today, build multi-tenant from day one — the
  whole business model is selling this to the next therapy/education center
  with zero code changes.

## 5. Database schema (draft)

```
tenants            id, name, slug, status(active|past_due|suspended|cancelled),
                   plan, paid_until, grace_days, created_at
tenant_settings    tenant_id, followup_hours, locale, ad_spend_monthly, ...
users              id (supabase auth), tenant_id, role(platform_admin|admin|operator),
                   full_name, phone, is_active
programs           id, tenant_id, name, type(therapy|course|internship),
                   price, currency, start_date, capacity, is_active
leads              id, tenant_id, full_name, phone, email,
                   source(fb_ads|ig_ads|manual|referral), meta_lead_id (unique),
                   program_id, stage(new|contacted|thinking|enrolled|lost),
                   stage_changed_at, assigned_to, notes, created_at
activities         id, tenant_id, lead_id, user_id,
                   type(note|call|email_sent|stage_change|reminder), payload jsonb,
                   created_at
email_templates    id, tenant_id, key, subject, body_html, locale, is_active
email_log          id, tenant_id, lead_id, template_key, status, provider_id, sent_at
reminders          id, tenant_id, lead_id, due_at, status(pending|done|snoozed),
                   created_by(system|user_id)
invoices           id, tenant_id, period, amount, currency,
                   status(unpaid|paid|void), paid_at        -- platform billing
platform_audit     id, actor_user_id, tenant_id, action, payload jsonb, created_at
feature_flags      tenant_id, flag, enabled
webhook_log        id, tenant_id, source, payload jsonb, status, created_at
```

Indexes: leads(tenant_id, stage), leads(meta_lead_id) unique, reminders(due_at, status),
activities(lead_id, created_at desc).

## 6. Key flows

### Meta Lead Ads → CRM
1. FB/IG lead form submitted → Meta sends `leadgen` webhook → Next.js route
   `/api/webhooks/meta` (verify signature) → fetch full lead via Graph API →
   upsert into `leads` (dedupe by meta_lead_id) → stage `new` → realtime
   appears on kanban → notification to operator.
2. Log every webhook in `webhook_log` for the platform health view.

### Follow-up engine (cron every 15 min)
- Find leads where stage='thinking' AND now() - stage_changed_at > tenant
  followup_hours AND no pending reminder → create reminder + notify operator;
  if tenant enabled auto-email → send follow-up template, log to email_log.

### Subscription enforcement (cron daily + middleware)
- Daily job: tenants where paid_until < now → past_due; past_due longer than
  grace_days → suspended.
- Next.js middleware on all tenant routes: load tenant status (cached);
  past_due → show banner; suspended → redirect to /payment-required.
  Platform routes unaffected.

## 7. Build phases

- **Phase 0 (day 1)**: repo, Supabase project, auth, tenant scaffold, RLS
  policies, CI/CD to Vercel, seed script.
- **Phase 1 — replaces the notebook (week 1–2)**: leads CRUD + manual add,
  kanban with dnd + realtime, lead detail with activity log, programs catalog,
  tenant user management. **Ship this to the client immediately.**
- **Phase 2 (week 3)**: Meta Lead Ads webhook integration, dedupe, source
  tracking, webhook log.
- **Phase 3 (week 4)**: follow-up engine, email templates + Brevo, email log,
  dashboard with conversion + ad ROI.
- **Phase 4 (week 5)**: platform super admin — tenants, invoices, suspend/
  reactivate, impersonation, audit log, feature flags, health view.
- **Phase 5 (later, upsell)**: corporate/B2B module, SMS (local GE gateway),
  online payment for subscriptions (Stripe or local provider), public booking
  form embeddable on their site.

## 8. Non-functional requirements

- RLS on 100% of tenant tables; no tenant data readable cross-tenant. Write a
  test that proves it.
- Webhook endpoint idempotent (Meta retries).
- All destructive actions soft-delete.
- Suspension never deletes or corrupts data.
- Impersonation always writes to platform_audit.
- Page loads < 1.5s on mid-range Android over 4G (the operator's reality).
- Daily DB backups (Supabase built-in on paid tier; pg_dump cron on free).
